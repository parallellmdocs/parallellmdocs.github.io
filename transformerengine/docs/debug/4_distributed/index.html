    分布式训练

    Nvidia-DL-Framework-Inspect 与 Transformer Engine 支持多 GPU 训练。本指南说明如何在分布式环境中使用调试工具，以及各功能在该环境下的行为。

    启用精度调试工具需满足：

    1. 在每个节点调用 debug_api.initialize(...)，并提供相同的配置 YAML 文件。
    2. 若需记录统计量，可调用 debug_api.set_tensor_reduction_group 指定适当的归约分组。

    功能行为

    在分布式场景下，DisableFP8GEMM 与 DisableFP8Layer 的行为与单 GPU 基本一致。

    PerTensorScaling 与 FakeQuant 会在每个节点独立计算 FP8 缩放因子，因此 GPU 数量会影响结果。这与延迟缩放（Delayed Scaling）FP8 配方不同，后者会同步缩放因子。

    图 1：对于 PerTensorScaling 与 FakeQuant，每个张量分片单独计算缩放因子；而延迟缩放 FP8 会在各节点间同步。

    与日志相关的功能更为复杂，后续章节将详细介绍。

    归约分组

    在张量并行、数据并行或流水线并行的设置中，部分张量会分布在多块 GPU 上，因此需要执行归约才能获得完整的统计量。

    权重张量始终按张量并行组划分，调试工具默认会在该组内自动归约。如需禁用，可调用：

    transformer_engine.debug.set_weight_tensor_tp_group_reduce(False)

    对于数据并行场景，Transformer Engine 模块本身不持有归约所需的进程组，可使用：

    debug_api.set_tensor_reduction_group(group)

    该调用保证统计量会在指定分组内归约。激活统计在前向结束（离开 autocast）后记录；梯度（dgrad、wgrad）统计在反向结束后记录。

    下图展示了 4 节点、张量并行度 2、数据并行度 2 的三种归约配置。

    图 2：所有节点组成单一的张量归约组，因此各节点记录到的统计量完全一致。

    图 3：每个节点将自身视为独立的归约组。权重统计仍在张量并行组内同步，但激活与梯度统计不再同步。

    图 4：通过 set_weight_tensor_tp_group_reduce(False) 禁用权重同步后，每个节点记录自身分片的权重统计。

    微批处理

    下面说明在微批（microbatch）场景下的统计行为。这里的微批指的是在一次 debug_api.step() 前执行多次 forward()。其表现为：

    • 对权重张量而言，统计量在每个微批间保持不变（权重未更新）。
    • 其他张量的统计量会累加。

    写入文件与 TensorBoard

    在单节点运行、且 default_logging_enabled=True 时，日志默认写入 log_dir/nvdlfw_inspect_statistics_logs/nvdlfw_inspect_globalrank-0.log。在多 GPU 训练中，每个节点会将归约后的统计写入自身的文件，同一归约组内的节点由于记录的是已归约数据，因此数值相同。

    若仅部分节点传入 TensorBoard writer，则只有这些节点会输出 TensorBoard 日志。这在复杂的并行场景（如有两个 Transformer 层，且 TP_SIZE=2、DP_SIZE=2、PP_SIZE=2）中非常有用。若希望所有统计写入 TensorBoard，需确保每个流水线并行组至少有一个进程传入 writer。

    图 5：流水线并行示例。为每个流水线并行组选择一个节点传入 tb_writer，并以此指定张量归约组。

    如果将张量归约组设置为 None，每个节点都会保留未归约的 wgrad/dgrad 统计，便于后续处理。若需禁用张量并行组内的权重归约，可调用：

    transformer_engine.debug.set_weight_tensor_tp_group_reduce(False)
