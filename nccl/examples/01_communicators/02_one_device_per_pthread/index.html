    NCCL 示例：pthread 一卡一线程

    本示例演示如何在 pthread 多线程场景中管理 NCCL 通信器，每个线程绑定一块 GPU。

    概述

    当应用本身采用线程并行时，可以为每个线程分配独立的 GPU 设备，并在各线程内完成通信器的初始化与销毁。本示例展示了正确的执行顺序。

    示例流程

    1. 创建线程
    2. 根据可见 GPU 数量（或 NTHREADS）创建等数量的 pthread；

    3. 每个线程设置自己的 CUDA 设备上下文。

    4. 创建通信器

    5. 通过 ncclCommInitRank 和共享的 unique ID 初始化通信器；

    6. 在线程内完成初始化，演示 NCCL 的线程安全用法。

    7. 验证信息

    8. 查询通信器的 rank、规模与设备；

    9. 确认所有线程都初始化成功。

    10. 清理资源

    11. 在线程内部按顺序销毁 NCCL 与 CUDA 资源；
    12. 演示正确的资源管理流程。

    构建与运行

    构建

    make [NCCL_HOME=<path-to-nccl>] [CUDA_HOME=<path-to-cuda>]

    指定线程（GPU）数量运行

    [NTHREADS=n] ./one_device_per_pthread

    打印 NCCL 调试信息

    NCCL_DEBUG=INFO ./one_device_per_pthread

    代码要点

    线程中的 ncclCommInitRank

    // 每个线程拥有自己的 threadData_t 结构体副本
typedef struct {
  int thread_id;      // 创建线程时指定
  int num_gpus;       // 通过查询 CUDA 设备得到
  ncclUniqueId commId; // 由 ncclGetUniqueId 生成
  ncclComm_t* comms;
} threadData_t;
threadData_t* data;

// 每个线程初始化对应的通信器
NCCLCHECK(ncclCommInitRank(&data->comms[thread_id], data->num_gpus, data->commId, data->thread_id));

    该方式下：

    • 每个线程获得独立的 NCCL rank（0、1、2…）；
    • unique ID 通过全局共享变量分发，无需额外通信。

    适用场景

    推荐使用

    • 线程化应用：程序本身利用线程并行；
    • 单节点任务：所有 GPU 位于同一台主机；
    • 共享内存需求：多个 GPU 上下文需要共享数据结构。

    不建议使用

    • 多节点集群：无法扩展到多节点；
    • 需要进程隔离：GPU 上下文必须隔离时；
    • 复杂系统：多进程架构可能更易维护。

    性能考量

    • 优势
    • 线程共享地址空间，数据交换方便；
    • 无 MPI 通信开销；
    • 适合快速实验。
    • 劣势
    • 线程同步逻辑更复杂；
    • 只能作用于单节点。

    常见问题

    1. 线程同步失败
    2. 确保所有线程使用同一个 NCCL unique ID；

    3. 合理使用 pthread 屏障与 pthread_join。

    4. CUDA 上下文冲突

    5. 每个线程在调用 CUDA API 前必须执行 cudaSetDevice()；

    6. 不要在不同线程之间共享 CUDA 流。

    7. 资源释放顺序

    8. 先销毁 NCCL 通信器，再释放 CUDA 资源；
    9. 在销毁通信器前确保流已同步。

    错误处理

    示例通过宏封装错误处理：

    • CUDACHECK：捕获 CUDA 错误并立即退出；
    • NCCLCHECK：捕获 NCCL 错误并立即退出；
    • 未进行异步错误检查，便于突出主流程；
    • 每个线程独立处理自身错误。

    相关环境变量

    • NTHREADS：创建线程数量（默认等于 GPU 数）。
