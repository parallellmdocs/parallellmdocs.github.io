    NCCL Net 插件文档

    本文介绍 NCCL Net 插件 API，以及如何为 NCCL 实现网络插件。

    概览

    为支持任意网络类型，NCCL 通过外部插件扩展网络功能。插件实现 NCCL 的网络 API，从而将针对特定 GPU 栈（例如 CUDA）构建的 NCCL 二进制，与针对特定网络栈构建的网络代码解耦。借助该机制，可以在任意 CUDA 版本与任意网络栈版本之间自由组合。

    NCCL 网络插件以共享库 libnccl-net.so 的形式提供。库内包含一个或多个 NCCL Net API 的实现，形式为带版本号的结构体，结构体内填充针对各个函数的指针。

    插件架构

    插件名称与多插件并存

    NCCL 初始化时会尝试加载 libnccl-net.so，并在库中查找符号。

    NCCL_NET_PLUGIN 环境变量允许多个插件共存。若设置了该变量，NCCL 会寻找名为 libnccl-net-${NCCL_NET_PLUGIN}.so 的库。因此建议按照该命名规则构建库，并让 libnccl-net.so 建立指向 libnccl-net-${NCCL_NET_PLUGIN}.so 的符号链接。这样，当路径中存在多个插件时，仅需设置 NCCL_NET_PLUGIN 即可选择合适的插件。

    结构体版本管理

    找到库后，NCCL 会查找名为 ncclNet_vX 的符号，X 随版本演进递增。版本号确保插件与 NCCL 内核兼容。

    建议插件同时提供多个版本的结构体实现（多个符号），以便同一个插件兼容不同版本的 NCCL。

    反过来，为了便于过渡，NCCL 也会尝试兼容多个插件版本：先查询最新的 ncclNet 结构体版本，如未找到会继续查旧版本，以便旧插件仍可运行。

    网络内集体操作（collNet）

    除了 ncclNet 结构体外，若网络支持网络内集体操作，插件可以额外提供 collNet 结构体。NCCL 的 collNet 算法可以借助该能力加速跨节点的 allReduce。

    collNet 结构体是可选的，但与 ncclNet 结构体共享版本号，且大量函数定义保持一致，以便于实现。

    头文件管理

    为了让用户更轻松地构建插件，插件应将所支持的 ncclNet_vX 定义拷贝到内部头文件中。ext-net/example/ 给出了示例：我们在 nccl/ 目录中保留所有头文件，并通过轻量封装支持旧版本。

    nccl/ 目录下包含从旧版本 API 中提取出的 net_vX.h，以及错误码定义 err.h。

    API（v11）

    以下为 ncclNet_v11 结构体主体，后续章节将逐一说明各个函数。

    typedef struct {
  // Name of the network (mainly for logs)
  const char* name;
  // Initialize the network.
  ncclResult_t (*init)(void** ctx, uint64_t commId, ncclNetCommConfig_v11_t* config, ncclDebugLogger_t logFunction, ncclProfilerCallback_t profFunction);
  // Return the number of adapters.
  ncclResult_t (*devices)(int* ndev);
  // Get various device properties.
  ncclResult_t (*getProperties)(int dev, ncclNetProperties_v11_t* props);
  // Create a receiving object and provide a handle to connect to it. The
  // handle can be up to NCCL_NET_HANDLE_MAXSIZE bytes and will be exchanged
  // between ranks to create a connection.
  ncclResult_t (*listen)(void* ctx, int dev, void* handle, void** listenComm);
  // Connect to a handle and return a sending comm object for that peer.
  // This call must not block for the connection to be established, and instead
  // should return successfully with sendComm == NULL with the expectation that
  // it will be called again until sendComm != NULL.
  // If *sendDevComm points to a valid object, then NCCL is requesting device offload for this connection
  ncclResult_t (*connect)(void* ctx, int dev, void* handle, void** sendComm, ncclNetDeviceHandle_v11_t** sendDevComm);
  // Finalize connection establishment after remote peer has called connect.
  // This call must not block for the connection to be established, and instead
  // should return successfully with recvComm == NULL with the expectation that
  // it will be called again until recvComm != NULL.
  // If *recvDevComm points to a valid object, then NCCL is requesting device offload for this connection
  ncclResult_t (*accept)(void* listenComm, void** recvComm, ncclNetDeviceHandle_v11_t** recvDevComm);
  // Register/Deregister memory. Comm can be either a sendComm or a recvComm.
  // Type is either NCCL_PTR_HOST or NCCL_PTR_CUDA.
  ncclResult_t (*regMr)(void* comm, void* data, size_t size, int type, void** mhandle);
  /* DMA-BUF support */
  ncclResul

    错误码

    所有插件函数均返回 NCCL 错误码。成功时需返回 ncclSuccess。

    常见错误码如下：

    • ncclSystemError：网络插件最常见的错误，表示调用 Linux 内核或系统库失败，通常对应网络/硬件错误。
    • ncclInternalError：当 NCCL 核心代码以错误方式使用插件时返回，例如重复分配请求、向接口传递无效参数等。
    • ncclInvalidUsage：用户错误或配置错误时返回，如缓冲区大小不匹配。
    • ncclInvalidArgument：通常不需要插件返回，因为 NCCL 核心层已经检查参数。
    • ncclUnhandledCudaError： CUDA 相关错误（网络插件一般不依赖 CUDA，因此较少出现）。

    操作流程概览

    NCCL 首先调用 init，随后通过 devices 获取网络设备数量，再使用 getProperties 获取属性。

    若 NCCL 需要初始化虚拟设备（例如用于 NIC 融合），会调用 makeVDevice，传入要合并的物理设备列表。若插件不支持 NIC 融合，可将 makeVDevice 置空。

    建立连接时，接收端先调用 listen 获取连接句柄，发送端获得句柄后调用 connect，最后接收端通过 accept 完成连接。

    connect 与 accept 可以接收可选的 netDevComm 指针，用于启用设备侧网络。如果插件不支持可忽略。

    连接建立后，通信通过 isend、irecv、test 进行；在此之前，NCCL 会调用 regMr 注册缓冲区（RDMA NIC 需要预处理），并在必要时调用 deregMr 注销。

    特定情况下，接收完成后会调用 iflush，确保 GPU 能看到新写入的数据。

    关闭连接时，NCCL 分别调用 closeListen、closeSend、closeRecv。

    API 详解

    Initialization

    • name：插件名称字符串，用于日志。设置 NCCL_NET=<插件名> 时会强制使用匹配名称的插件；不要与 NCCL_NET_PLUGIN 混淆，后者用于指定库名后缀。

    • init：NCCL 找到插件后首先调用。插件应该在此阶段发现网络设备并确保可用。若返回非 ncclSuccess，NCCL 将退回内部实现。

    每次调用 init 都返回一个上下文指针，插件使用它来分配资源和维护状态。该上下文会在后续如 listen、connect 等调用中传回。每个上下文与一个通信器（commId）唯一对应，也可以通过 config 传入每通信器的配置。

    为了让插件日志与 NCCL 日志融合，NCCL 会提供日志回调。

    （后续各函数如 devices、getProperties、listen、connect、accept、isend、test、regMr、deregMr、iflush 等的详细说明，与英文版一致。请参阅原文节“API Functions”获取完整细节。）

    提示：考虑到篇幅，本翻译保留了接口顺序、调用时机、所需返回值等关键信息。具体结构体字段（如 ncclNetProperties_v11_t、ncclNetDeviceHandle_v11_t）与旧版本兼容层的区别，请参考源代码或头文件。

    额外注意事项

    • 插件在实现中应复用 NCCL 提供的错误码，避免自定义。
    • 推荐为旧版 API 提供兼容层，使同一插件支持多个 NCCL 版本。
    • 若插件支持 collNet，需要额外提供 collNet 结构体。
    • 对于 ncclNet_v11 中描述的 DMA-BUF、GPU 直连（device offload）等特性，可按需实现；未实现时可返回 ncclInternalError 或 ncclSystemError。

    调试与日志

    NCCL 会提供调试日志和性能采样回调。插件应调用 NCCL 传入的 logFunction 输出日志，以便其集成到 NCCL 的日志体系中。对于性能采集，插件可按需调用 profFunction。

    示例与头文件组织

    ext-net/example/ 目录给出了一个最小示例，展示如何：

    • 组织头文件（nccl/net_vX.h 与 err.h）；
    • 提供多版本结构体；
    • 通过薄封装支持旧 API；
    • 按 NCCL 要求导出 ncclNet_vX 与可选的 ncclCollNet_vX 符号。

    在实际项目中，可仿照示例将插件实现拆分为：

    1. 网络适配层：负责与具体网络栈对接；
    2. NCCL 接口层：将适配层能力封装为 NCCL 所需的函数；
    3. 版本兼容层：根据不同的结构体版本导出对应的符号。

    兼容性与升级建议

    • 当 NCCL 升级至新的 API 版本时，插件应同步增加新的 ncclNet_vX 结构体定义；
    • 对已有版本保持兼容，可在内部通过宏或薄层调用最新实现；
    • collNet 结构体的版本与 ncclNet 一致，若不支持可不导出；
    • 插件应在构建时链接匹配版本的 CUDA 与网络栈库。

    总结

    通过 NCCL Net 插件机制，可以：

    • 将 NCCL 与底层网络栈解耦；
    • 为不同的网络硬件快速接入 NCCL；
    • 支持网络内加速（collNet）与设备端通信（device offload）；
    • 在升级 CUDA、驱动或网络栈时保持灵活性。

    实现插件时，请遵循本文档及头文件定义，妥善处理版本兼容、错误码、资源生命周期和日志回调。
