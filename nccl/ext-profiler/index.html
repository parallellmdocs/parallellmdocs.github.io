    NCCL Profiler 插件文档

    本文介绍 NCCL Profiler 插件 API，以及如何为 NCCL 实现 Profiler 插件。

    概览

    为便于与深度学习框架集成，NCCL 自 v2.23 起提供了 Profiler 插件接口。任何 NCCL 用户都可以编写 Profiler 插件，从 NCCL 中提取性能数据，用于调试与分析。

    和其他插件（如网络插件）类似，Profiler 插件以共享库 libnccl-profiler.so 的形式提供。库中包含一个或多个带版本号的结构体，实现 NCCL Profiler API，并为各个函数提供指针。

    插件架构

    插件名称与多插件并存

    NCCL 初始化时会尝试加载 libnccl-profiler.so 并解析符号。

    NCCL_PROFILER_PLUGIN 环境变量允许多个插件共存。设置后，NCCL 会寻找名为 libnccl-profiler-${NCCL_PROFILER_PLUGIN}.so 的库。因此推荐按该模式命名库，同时创建从 libnccl-profiler.so 到对应库的符号链接。若路径中存在多个插件，只需设置 NCCL_PROFILER_PLUGIN 即可选择需要的插件；也可直接将 NCCL_PROFILER_PLUGIN 设置为库的绝对路径。

    结构体版本管理

    找到库后，NCCL 会查找 ncclProfiler_vX 符号，X 随版本递增。版本号确保插件与 NCCL 核心兼容。

    建议插件同时导出多个版本的结构体符号，使其兼容不同版本的 NCCL。

    同样地，为方便过渡，NCCL 会按版本从高到低查找结构体，确保旧插件依然可用。

    头文件组织

    为简化构建流程，插件应将所支持的 ncclProfiler_vX 定义拷贝到自身头文件中。ext-profiler/example 给出了示例：所有头文件放在 nccl/ 目录，并通过薄封装兼容旧版本。

    nccl/ 目录包含 profiler_vX.h（从旧版 API 中提取的定义）以及错误码 err.h。

    API（v5）

    以下为 ncclProfiler_v5 结构体，后文将逐一说明各函数。

    typedef struct {
  const char* name;
  ncclResult_t (*init)(void** context, uint64_t commId, int* eActivationMask, const char* commName, int nNodes, int nranks, int rank, ncclDebugLogger_t logfn);
  ncclResult_t (*startEvent)(void* context, void** eHandle, ncclProfilerEventDescr_v5_t* eDescr);
  ncclResult_t (*stopEvent)(void* eHandle);
  ncclResult_t (*recordEventState)(void* eHandle, ncclProfilerEventState_v5_t eState, ncclProfilerEventStateArgs_v5_t* eStateArgs);
  ncclResult_t (*finalize)(void* context);
} ncclProfiler_v5_t;

    错误码

    原则上，Profiler 不应将自己的错误上报给 NCCL 更改流程。虽然接口返回 NCCL 错误码，但除 init 外应始终返回 ncclSuccess。若 init 返回错误，NCCL 将禁用该插件。

    执行流程概览

    每个新通信器初始化时，NCCL 会调用 init，插件应返回一个上下文句柄，用于隔离不同通信器的 Profiler 实例；通信器销毁时会调用 finalize，释放资源。

    NCCL 核心在关键路径中插入了 startEvent、stopEvent 与 recordEventState，分别用于创建事件、停止事件、记录状态。

    API 说明

    初始化

    • name：插件名称字符串，用于日志（特别是在 NCCL_DEBUG=INFO 时）。
    • init：发现插件后，NCCL 会调用 init 初始化内部上下文。若返回值不是 ncclSuccess，NCCL 将禁用插件。
    • finalize：释放上下文，回收资源。

    Profiling

    • startEvent：开始记录一个事件。参数包含上下文、事件描述符 ncclProfilerEventDescr_t，并返回一个事件句柄（供后续使用）。

    事件描述符包含事件元数据，不同事件类型拥有各自的字段。ncclProfilerEventDescr_v5_t 定义如下：

    typedef struct {
  uint64_t type;
  void* parentObj;
  int rank;
  union {
    struct { bool graphCaptured; int groupDepth; } groupApi;
    struct { const char* func; size_t count; const char* datatype; int root; void* stream; bool graphCaptured; } collApi;
    struct { const char* func; size_t count; const char* datatype; void* stream; bool graphCaptured; } p2pApi;
    struct { void* stream; } kernelLaunch;
    struct { uint64_t seqNumber; const char* func; const void* sendBuff; void* recvBuff; size_t count; int root; const char* datatype; uint8_t nChannels; uint8_t nWarps; const char* algo; const char* proto; void* parentGroup; } coll;
    struct { const char* func; void* buff; const char* datatype; size_t count; int peer; uint8_t nChannels; void* parentGroup; } p2p;
    struct { pid_t pid; uint8_t channelId; int peer; int nSteps; int chunkSize; int isSend; } proxyOp;
    struct { int step; } proxyStep;
    struct { uint8_t channelId; uint64_t ptimer; } kernelCh;
    struct { int64_t id; void* data; } netPlugin;
  };
} ncclProfilerEventDescr_v5_t;

    NCCL 定义的事件类型包括：ncclProfileGroupApi、ncclProfileCollApi、ncclProfileP2pApi、ncclProfileKernelLaunch、ncclProfileGroup、ncclProfileColl、ncclProfileP2p、ncclProfileProxyOp、ncclProfileProxyStep、ncclProfileProxyCtrl、ncclProfileKernelCh、ncclProfileNetPlugin。

    • stopEvent：停止事件，参数为 startEvent 返回的句柄。停止后句柄不可再用于其他接口。

    • recordEventState：更新事件状态与属性。部分事件（如 ncclProfileP2pApi、ncclProfileCollApi 等）仅支持 start/stop；而 ncclProfileGroupApi、ncclProfileProxyOp、ncclProfileProxyStep、ncclProfileNetPlugin、ncclProfileKernelCh、ncclProfileProxyCtrl 可通过该函数更新状态。

    支持的状态枚举如下：

    typedef enum {
  // ncclProfileProxyOp event states
  ncclProfilerProxyOpSendPosted        = 0,  // v4 起弃用
  ncclProfilerProxyOpSendRemFifoWait   = 1,  // v4 起弃用
  ncclProfilerProxyOpSendTransmitted   = 2,  // v4 起弃用
  ncclProfilerProxyOpSendDone          = 3,  // v4 起弃用
  ncclProfilerProxyOpRecvPosted        = 4,  // v4 起弃用
  ncclProfilerProxyOpRecvReceived      = 5,  // v4 起弃用
  ncclProfilerProxyOpRecvTransmitted   = 6,  // v4 起弃用
  ncclProfilerProxyOpRecvDone          = 7,  // v4 起弃用
  ncclProfilerProxyOpInProgress_v4     = 19, // 标记 proxy op 进入 progress 状态

  // ncclProfileProxyStep event states
  ncclProfilerProxyStepSendGPUWait     = 8,
  ncclProfilerProxyStepSendPeerWait_v4 = 20,
  ncclProfilerProxyStepSendWait        = 9,
  ncclProfilerProxyStepRecvWait        = 10,
  ncclProfilerProxyStepRecvFlushWait   = 11,
  ncclProfilerProxyStepRecvGPUWait     = 12,

  // ncclProfileProxyCtrl event states
  ncclProfilerProxyCtrlIdle            = 13,
  ncclProfilerProxyCtrlActive          = 14,
  ncclProfilerProxyCtrlSleep           = 15,
  ncclProfilerProxyCtrlWakeup          = 16,
  ncclProfilerProxyCtrlAppend          = 17,
  ncclProfilerProxyCtrlAppendEnd       = 18,

  // ncclProfileNetPlugin event states
  ncclProfilerNetPluginUpdate          = 21,

    （为简洁起见，省略了同类状态的详细注释；具体含义与英文原文一致。）

    状态更新可能伴随属性变化，ncclProfilerEventStateArgs_v5_t 用于携带这些附加信息：

    typedef union {
  struct { size_t transSize; } proxyStep;
  struct { int appendedProxyOps; } proxyCtrl;
  struct { void* data; } netPlugin;
  struct { uint64_t pTimer; } kernelCh;
} ncclProfilerEventStateArgs_v5_t;

    示例插件 ext-profiler/example 展示了如何捕获并使用上述事件。

    事件层级

    NCCL 事件构成如下层级：

    Group API
 ├─ Collective API
 │   ├─ Collective
 │   │   ├─ ProxyOp
 │   │   │   └─ ProxyStep
 │   │   │       └─ NetPlugin
 │   │   └─ KernelCh
 │   └─ (更多子事件)
 ├─ Point-to-point API
 │   ├─ Point-to-point
 │   │   ├─ ProxyOp
 │   │   │   └─ ProxyStep
 │   │   │       └─ NetPlugin
 │   │   └─ KernelCh
 └─ Kernel Launch

ProxyCtrl（独立）

    Profiler 插桩与日志

    集体与点对点

    NCCL 在不同层面注入回调，以捕获 group、collective 与 p2p 操作，以及 proxy、kernel、network 的活动。由于 NCCL 操作是异步的，仅通过 API 事件难以确定结束时间。因此，对于集体操作，stopEvent 仅表示操作已入队；若需要获知真实完成时刻，可结合 proxy 或 kernel 事件判断，例如使用最后一个 ncclProfileProxyOp 的 stopEvent 来标记完成。常见做法是对事件进行引用计数，在 startEvent/stopEvent 中增加/减少计数。

    PXN

    使用 PXN 时，某些 proxy 操作会在远端线程执行，导致事件层级跨进程。由于事件描述符包含父事件指针，必须确保远端线程与原线程位于同一地址空间，否则访问父事件将非法。为此，事件描述符包含原始 PID，插件需要确认 PID 一致后再访问父事件。

    已知限制

    在纯节点内通信或无网络活动时，Profiler 只能看到入队事件（如 ncclAllReduce），时间戳仅代表入队时刻。若想获得真实执行时间，需要启用 proxy 与 kernel 事件。

    自 v3 起，Profiler 不再强制依赖网络事件进行节点内分析；内核事件通过计数器暴露给主机与 proxy。若 proxy 正在处理网络请求或 CPU 负载较高，采样可能延迟。

    为缓解该问题，自 v4 起 NCCL 为每个通道维护 64 槽的环形缓冲区，并提供由内核返回的起止时间戳（ptimer），NCCL 会将其传给 Profiler 插件，插件可转换为 CPU 时间域。
