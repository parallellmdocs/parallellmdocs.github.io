    context_parallel 模块

    Context parallelism 概述

    图 1：Transformer 层在 TP2CP2 配置下的运行示意。Attention 旁边的通信属于 CP，其余属于 TP。（AG/RS：前向 all-gather、反向 reduce-scatter；RS/AG：前向 reduce-scatter、反向 all-gather；/AG：前向空操作、反向 all-gather）。

    Context Parallelism（简称 “CP”）是一种针对序列长度维度的并行方案。与先前的序列并行（SP）仅拆分 Dropout 与 LayerNorm 激活不同，CP 会沿序列维度切分网络输入及所有激活。使用 CP 时，除注意力模块外的其他算子（如线性层、LayerNorm 等）均无需修改，因为它们没有跨 token 的操作。对于注意力而言，每个 token 的 Q（query）需要与同一序列中所有 token 的 K/V（key/value）计算，因此 CP 需要在多卡之间额外执行 all-gather 来收集完整的 KV 序列，并在反向传播时对 KV 的激活梯度执行 reduce-scatter。为了降低激活占用，每张 GPU 在前向阶段只保存自己负责的序列段对应的 KV，并在反向时重新聚合。KV 的通信发生在不同张量并行组的对应 GPU 之间，其 all-gather 与 reduce-scatter 在底层被转换为环形拓扑上的点对点通信。若采用 MQA/GQA 等方案，只需少量注意力头即可维护 KV，从而进一步降低通信量。

    例如图 1 中，假设序列长度为 8K，每张 GPU 负责 4K 个 token。GPU0 与 GPU2 组成一个 CP 组，需要互相交换 KV；GPU1 与 GPU3 亦然。CP 与 Ring Attention 相似，但通过以下方式取得更好的性能：(1) 利用最新的开源与 cuDNN flash attention 内核；(2) 去除因下三角因果 mask 带来的冗余计算，实现最优负载均衡。

    Context parallelism 的收益

    图 2：175B GPT 在不同 TP+CP 组合下相对于全重计算（TP8CP1）的加速比。

    当上下文长度增长时，大语言模型的激活内存线性增加，容易遭遇 OOM。虽然可以在反向阶段重计算激活避免 OOM，但开销约为 30%。将张量并行（TP）增大也能缓解 OOM，但会使算子更短，从而难以掩蔽通信延迟；无论是否发生 OOM，扩大 TP 都可能遇到这一问题。

    CP 能更好地解决上述瓶颈。借助 CP，每张 GPU 仅处理序列的一部分，计算与通信量都按 CP 倍数下降，因此二者之间不会出现重叠问题。同时，每张 GPU 上的激活占用也按 CP 倍数缩小，从根源上避免 OOM。如图 2 所示，合理组合 TP 与 CP 可以在不引入重计算开销的情况下，实现计算与通信的最佳平衡。

    如何启用 context parallelism

    GPT 已内置 CP 支持，共享 GPT 代码路径的模型（例如 Llama）同样受益。CP 可与 TP（张量并行）、PP（流水线并行）、DP（数据并行）协同工作，总 GPU 数为 TP × CP × PP × DP。CP 还兼容多种注意力变体，包括 MHA、MQA、GQA，以及单向或双向掩码。

    在命令行设置 context_parallel_size= 即可启用 CP。默认值为 1，表示关闭 CP。启用 CP 需要 Megatron-Core 版本不低于 0.5.0，Transformer Engine 版本不低于 1.1。
