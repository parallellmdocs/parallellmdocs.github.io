    Megatron Core MoE

    Megatron-Core MoE 提供完整的并行策略，将专家并行无缝融合到张量、数据、序列与流水线并行中。在 MCore v0.9 中，我们已实现 468 TFLOPS 的 Mixtral 8x7B bf16 训练性能，并支持 DeepSeek-V3、Qwen-MoE 等最新的 MoE 架构。

    最新特性

    • 支持 DeepSeek-V3 架构
    • 为 MLA 与 DeepSeek-V3 启用 TP
    • 为 MLA 与 DeepSeek-V3 启用 CP
      • 需要 TransformerEngine >= 2.5.0
      • 特别感谢来自小红书的 SuperCB 和腾讯微信基础设施团队的 RandMist 的贡献
    • 支持无辅助损失（aux-loss-free）的负载均衡策略
    • 支持 node-limited routing
    • 支持 Multi-Token Prediction (MTP)
    • 批级重叠以隐藏 EP-A2A 通信
    • 集成 DeepSeek 的 DeepEP，实现高效 token 分发与聚合
    • 新增 token permutation/unpermutation 融合
    • 支持不均匀的虚拟流水线并行拆分
    • 支持在部分子模块上丢弃输出的检查点策略

    并行策略

    • 专家并行（Expert Parallelism）
      • 针对 MoE 模型的专用并行方式，专家被切分到不同 worker，每个 worker 处理不同批次的训练样本，并在每个 MoE 层上负责一个或多个专家。
    • 3D 并行：数据并行、张量并行、流水线并行
      • 注意：当同时使用专家并行与张量并行时，必须启用序列并行（sequence parallelism）。
    • 上下文并行（Context Parallelism）
      • 拆分序列维度，支持长上下文训练。
    • 更丰富的映射方式：EP 可与 DP/TP/PP/CP 组合，以应对更大型的 MoE 变体。
    • MoE Parallel Folding：可为注意力与 MoE 组件设置不同的并行策略，实现更加灵活高效的模型切分，详见下文。
    • 全面支持分布式优化器。

    路由与负载均衡

    • 路由类型：
      • Top-K MLP router
    • 负载均衡算法：
      • Sinkhorn（S-BASE）
      • 辅助损失 / 负载均衡损失
      • 无辅助损失的负载均衡策略
    • CUDA 融合路由与负载均衡内核

    性能优化

    • （实验特性）集成 DeepEP，用于大规模 MoE 训练中的高效 token 通信。
    • 当本地专家数 > 1 时启用 GroupedGEMM
      • 支持 dtype：bf16
      • 提升大型 MoE 模型性能
    • 为 MoE 启用 --tp-comm-overlap
    • 支持 FP8 训练

    Token 分发机制

    • 无丢弃 / Dropless
    • Token 丢弃，可选择是否填充至容量
    • Token permutation / unpermutation 融合

    易用性

    • 提供 Mixtral 模型的检查点转换工具，详见 示例。
    • MoE 层频率（MoE Layer Frequency）可自定义混合的 MoE/Dense 架构。
    • 支持分布式检查点
    • 支持按层日志
    • 支持 Upcycling

    使用指南

    使用方法

    快速上手

    若要训练一个 top-2、包含 8 个专家并启用辅助损失的 MoE 模型，可添加以下参数：

    --num-experts 8
--expert-model-parallel-size 8
--moe-grouped-gemm
--moe-permute-fusion
--moe-router-load-balancing-type aux_loss # Options: aux_loss, sinkhorn, none (default: aux_loss)
--moe-router-topk 2
--moe-aux-loss-coeff 1e-2
--use-distributed-optimizer
--moe-token-dispatcher-type alltoall

    若要启用类 GShard、SwitchTransformer 的 token drop 机制，可额外设置：

    --moe-expert-capacity-factor 1.0
--moe-pad-expert-input-to-capacity # Optional

    下图展示了 MCore 中不同的丢弃策略：

    1. 默认的 dropless 策略不会丢弃或填充 token。
    2. 设置 --moe-expert-capacity-factor 后，超过容量的 token 会按选中的概率被丢弃。若 EP > 1，丢弃发生在 EP rank 之间进行 token 交换之前。容量计算公式为 capacity = num_tokens_per_rank * topk * capacity_factor / num_experts。
    3. 设置 --moe-pad-expert-input-to-capacity 后，token 少于容量的专家会自动填充到容量。

    微调 Mixtral 模型

    Megatron-Core 完全支持 Mixtral MoE 模型，并提供从 HuggingFace 格式转换为 MCore 格式的检查点工具。

    分布式检查点

    MCore v0.7 引入了完全并行且异步的保存能力，可在保存期间与训练计算重叠，显著缩短训练暂停时间。

    MoE Parallel Folding

    MoE Parallel Folding 允许你为注意力与 MoE 部分设置不同的并行策略。通过 --transformer-impl transformer_engine --moe-enable-moe-pf 等参数，可让注意力层与 MLP/专家层采用不同的映射方式，从而获得更灵活的模型切分。例如在 DeepSeek-V3 中，可将注意力映射到（TP, PP），而 MoE 则映射到（EP, CP, PP）。

    混合 MoE/Dense 架构

    MoE 层频率允许在全模型中交替插入 Dense 层与 MoE 层，通过 --moe-layer-freq 指定周期。例如设置为 4 表示每 4 层 Transformer 插入一个 MoE 层，其余为 Dense 层，可构建混合架构。

    批级 EP-A2A 重叠

    启用批级的 All-to-All 重叠（灵感来自 DeepSeek-V3 的 DualPipe 实现）可通过 --moe-batch-level-a2a-overlap 等参数实现。其特性包括： - 通过批级重叠隐藏 EP A2A 通信 - 将权重梯度与激活梯度计算拆分，以便与通信更好地重叠 - 支持交错流水线并行 - 支持 FP8 训练 - 支持 MTP（目前仅支持 --mtp-num-layers 1）

    使用方式： 

    # Add the following flags to the training script
--moe-batch-level-a2a-overlap
--pipeline-parallel-schedule-lbfs
--num-lbfs-batches 8

    Token Dispatcher 类型

    Megatron-Core 支持多种 token dispatcher： - alltoall：经典的 All-to-All token 分发 - flex：支持 DeepEP 的灵活分发，实现跨节点高性能通信

    通过 --moe-token-dispatcher-type 指定，若搭配 --moe-enable-deepep 则启用 DeepEP。

    DeepSeek 的 DeepEP

    • DeepSeek-DeepEP 提供针对大规模 MoE 训练优化的 token 分发与聚合实现。
    • 特别适合 DeepSeek-V3 等大规模细粒度 MoE 架构。
    • 启用方式：在命令中添加 --moe-token-dispatcher-type=flex --moe-enable-deepep。

    CUDA Graph 支持

    通过 --cuda-graph-impl 参数启用 CUDA Graph，有两种实现：

    1. --cuda-graph-impl=local：使用 MCore 内部的 CUDA Graph 管理器。
    2. --cuda-graph-impl=transformer_engine：使用 TransformerEngine 的 make_graphed_callables() 接口。

    使用第二种方式需在训练脚本中调用 TECudaGraphHelper.create_cudagraphs() 与 TECudaGraphHelper.cuda_graph_set_manual_hooks()，示例见 megatron/training/training.py。

    对于 MoE 模型，如果未设置 --moe-expert-capacity-factor 与 --moe-pad-expert-input-to-capacity，其动态形状会阻止对 MoE 层进行 CUDA Graph 捕获。此时可以通过 --cuda-graph-scope=attn 只对注意力部分（TransformerLayer._forward_attention()）启用 CUDA Graph，而让 MoE 部分（TransformerLayer._forward_mlp()）保持原状。

    Checkpointing

    新增的“丢弃输出”检查点策略允许在前向过程中丢弃特定子模块的输出，并在反向阶段重新计算，以节省内存。可通过 --recompute-granularity selective --recompute-modules [submodule1, submodule2, ...] 启用，支持的子模块包括：

    • moe_act：重新计算 GroupedMLP 激活函数
    • layernorm：重新计算 input_layernorm 与 pre_mlp_layernorm（在它们不是 IdentityOp 时）
    • mla_up_proj：重新计算 MLA 上投与 RoPE 应用部分
    • core_attn：重新计算核心注意力子模块（使用标准检查点方式）
    • mlp：重新计算稠密 MLP 子模块（使用标准检查点方式），适合 DeepSeek-V3 等混合模型
    • moe：重新计算 MoE 层子模块（使用标准检查点方式）

    Upcycling

    通过 --moe-use-upcycling 可启用 upcycling：从 --load 目录加载稠密模型，在运行时转换为 MoE 模型并开始训练。转换后的模型会在训练前保存至 --save 路径。该能力基于分布式检查点构建，支持与原稠密检查点不同的并行模式，训练时可使用任意专家并行度。

    此外还支持在最新研究成果基础上的更细粒度 upcycling 策略（详见论文）。默认策略是将现有 MLP 复制为多个专家；细粒度策略通过 --moe-upcycling-granularity 指定专家隐藏层相对于原稠密 FFN 的缩放倍数。将该参数设为正整数即可启用；若设为 1，则等同于默认策略。使用细粒度策略时，MoE 的 FFN 隐藏维度应为稠密 FFN 隐藏维度除以 --moe-upcycling-granularity。

    注意：MoE 模型结构通过脚本参数定义。所有 MoE 相关参数（如 --num-experts）都可自定义，但其余结构参数需与原稠密模型保持一致。

    性能建议

    • 当本地专家数较大时，请启用 --moe-grouped-gemm 与 --moe-permute-fusion。
    • 使用 --moe-router-load-balancing-type sinkhorn 可获得更均衡的 token 分配，但计算开销更高。
    • 为减少通信瓶颈，建议搭配 --tp-comm-overlap 与批级 A2A 重叠。
    • 对于长序列任务，可结合上下文并行（--context-parallel-size）与专家并行获取更高效率。

    监控与调试

    • 可通过 --log-throughput 与 --log-layerwise-times 等参数监控 MoE 层性能。
    • 启用 --capture-environment-metrics 可记录 GPU 功耗等信息，辅助定位性能瓶颈。
