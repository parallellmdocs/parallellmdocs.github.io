    Multi-Storage Client (MSC) 集成

    Multi-Storage Client（MSC）提供统一接口，可从文件系统（例如本地磁盘、NFS、Lustre）以及对象存储（如 S3、GCS、OCI、Azure、AIStore、SwiftStack）读取数据集并存储检查点。

    本指南将帮助你完成以下任务：

    1. 安装并配置 MSC
    2. 直接使用对象存储中的数据集训练模型
    3. 在对象存储中保存与加载模型检查点

    安装

    MSC 在 PyPI 上以 multi-storage-client 包提供。

    基础 client 默认支持 POSIX 文件系统；针对每种存储服务还提供额外依赖包，以便访问对应的对象存储。

    # POSIX 文件系统
pip install multi-storage-client

# AWS S3 及兼容对象存储
pip install "multi-storage-client[boto3]"

# Google Cloud Storage (GCS)
pip install "multi-storage-client[google-cloud-storage]"

    配置文件

    MSC 使用 YAML 配置文件定义与对象存储的连接方式。你可以在其中声明一个或多个存储 profile，每个 profile 都对应一个存储后端或桶。通过集中管理配置，训练脚本无需硬编码密钥、桶名或其他供应商参数，就能保持整洁与可移植性。

    示例配置如下：

    profiles:
  my-profile:
    storage_provider:
      type: s3
      options:
        # base_path 设为桶/容器名称
        base_path: my-bucket
        region_name: us-west-2
    # 可选凭据（S3 也可通过环境变量获取）
    credentials_provider:
      type: S3Credentials
      options:
        access_key: ${AWS_ACCESS_KEY}
        secret_key: ${AWS_SECRET_KEY}

cache:
  size: 500G               # 缓存最大容量
  location: /tmp/msc_cache # 缓存目录

    在运行 Megatron-LM 脚本前，通过环境变量指定配置文件位置：

    export MSC_CONFIG=/path/to/msc_config.yaml

    MSC URL 格式

    MSC 使用自定义 URL 方案来访问不同对象存储中的文件，格式如下：

    msc://<profile-name>/<path/to/object>

    组成部分：

    • msc://：表示该路径由 MSC 解析。
    • <profile-name>：对应 YAML 配置中 profiles 下的 profile 名称，包含存储类型、凭据、基础路径等信息。
    • <path/to/object>：相对于 profile 中 base_path 的对象或目录路径，类似本地文件系统路径，但在底层映射到对象存储的键。

    示例：

    profiles:
  my-profile:
    storage_provider:
      type: s3
      options:
        base_path: my-bucket

    使用以下 MSC URL：

    msc://my-profile/dataset/train/data.bin

    MSC 会访问 S3 桶 my-bucket 中键为 dataset/train/data.bin 的对象。如果改为 GCS 或 OCI profile，MSC 会自动切换至对应后端逻辑，而应用代码无需改动。

    借助这一抽象，训练脚本只需切换配置文件中的 profile，即可在 AWS、GCP、Oracle、Azure 等环境之间统一访问资源。

    使用对象存储训练

    将 MSC URL 与 --data-path 参数结合，即可访问存储在对象存储中的数据集。此外，Megatron-LM 在读取对象存储时需要指定 --object-storage-cache-path，用于缓存 IndexedDataset 的 .idx 文件。

    python pretrain_gpt.py                                      \
    --object-storage-cache-path /path/to/object_store_cache \
    --data-cache-path /path/to/data_cache                   \
    --data-path msc://my-profile/datasets/text_document     \
    --no-mmap-bin-files

    注意： 使用 MSC 读取对象存储数据集时，上述四个参数都必不可少。

    在对象存储中保存与加载检查点

    通过在 --save 与 --load 参数中使用 MSC URL，可直接在对象存储中读写模型检查点：

    python pretrain_gpt.py                \
  --save msc://my-profile/checkpoints \
  --load msc://my-profile/checkpoints \
  --save-interval 1000

    注意： 当前仅在使用 MSC URL 时支持 torch_dist 检查点格式。

    禁用 MSC

    安装 multi-storage-client 后，MSC 集成功能默认启用。即使路径为本地文件系统（例如 --data-path、--save 或 --load 传入 /filesystem_mountpoint/path），也会通过 MSC 处理。该抽象层对常规路径的性能影响极小，因此通常无需禁用。如需关闭，可使用 --disable-msc：

    python pretrain_gpt.py --disable-msc

    性能注意事项

    使用 MSC 访问对象存储时应关注以下性能影响：

    读取数据集

    直接从对象存储读取训练数据通常比本地磁盘更慢，主要原因包括： * 对象存储在小块或随机读取时延迟较高（例如访问 .bin 样本）。 * 对象存储使用基于 HTTP 的协议（如 S3 GET + range 请求），速度低于本地文件系统 I/O。

    为弥补延迟，建议通过 --num-workers 参数增加数据加载进程数：

    python pretrain_gpt.py --num-workers 8 ...

    更多 worker 能实现更高的并行读取，以掩盖 I/O 延迟并保持 GPU 高利用率。

    加载检查点

    使用 MSC 从对象存储加载检查点时，应在 MSC 配置中启用缓存，以缓存下载的检查点数据与元数据，从而显著缩短加载时间并降低内存占用。例如：

    cache:
  size: 500G
  location: /tmp/msc_cache

    建议将缓存目录配置在高速本地存储（如 NVMe SSD）上。

    更多资源与高级配置

    完整配置选项请参考 MSC 配置文档，其中包含关于支持的存储提供方、凭据管理与高级缓存策略的详细说明。

    MSC 支持采集可观测性指标与追踪，以帮助监控、调试训练过程中的数据访问模式，识别数据管线瓶颈、优化缓存策略，并在使用对象存储训练大规模数据集时监控资源利用率。更多信息见 MSC 可观测性文档。

    MSC 还提供实验性的 Rust 客户端，可绕过 Python GIL，在多线程 I/O 操作中显著提升性能。该客户端支持 AWS S3、SwiftStack 与 Google Cloud Storage，实现真正的并发执行。若要启用，在存储提供方配置中添加 rust_client: {}。详情见 MSC Rust 客户端文档。
