    分布式通信包 —— torch.distributed

    :::{note} 如需快速了解分布式训练的全部功能，请参阅 PyTorch 分布式概览。 :::

    .. automodule:: torch.distributed

    .. currentmodule:: torch.distributed

    后端

    PyTorch 自带的后端

    PyTorch 分布式包在 Linux（稳定版）、macOS（稳定版）和 Windows（原型）上受支持。 在 Linux 上默认构建并包含 Gloo 和 NCCL 后端（仅当启用 CUDA 时才构建 NCCL）。MPI 属于可选后端，需要从源码编译 PyTorch 时手动启用（例如在安装了 MPI 的主机上构建）。

    :::{note} 自 PyTorch v1.8 起，Windows 支持除 NCCL 之外的所有通信后端。 如果 {func}init_process_group 的 init_method 参数指向某个文件，该文件路径必须符合以下模式：

    • 本地文件系统：init_method="file:///d:/tmp/some_file"
    • 共享文件系统：init_method="file://////{machine_name}/{share_folder_name}/some_file"

    与 Linux 平台相同，可以通过设置环境变量 MASTER_ADDR 和 MASTER_PORT 启用 TcpStore。 :::

    选择哪个后端？

    某些后端只支持特定的部署环境。例如：Gloo 适用于多 CPU 训练，NCCL 支持多 GPU 训练，MPI 则需要正确配置的 MPI 实现。需要同时考虑可用硬件、部署环境以及运维成本。

    网络接口

    在拥有多个网络接口的环境（例如多个网卡或 Infiniband）中，需要显式设置以下环境变量：

    • NCCL_SOCKET_IFNAME，例如 export NCCL_SOCKET_IFNAME=eth0
    • GLOO_SOCKET_IFNAME，例如 export GLOO_SOCKET_IFNAME=eth0

    对于 Gloo，可以使用逗号分隔多个接口，例如 export GLOO_SOCKET_IFNAME=eth0,eth1,eth2,eth3。 Gloo 会以轮询方式在这些接口之间分发操作。所有进程必须保证在该变量中指定相同数量的接口。

    其他 NCCL 环境变量

    调试 —— 当 NCCL 出现错误时，可设置 NCCL_DEBUG=INFO 输出警告信息和初始化日志。

    也可以使用 NCCL_DEBUG_SUBSYS 定位特定子系统。例如，NCCL_DEBUG_SUBSYS=COLL 会打印 集合通信日志，便于排查通信类型或消息大小不匹配导致的阻塞；若拓扑检测失败，可以设置 NCCL_DEBUG_SUBSYS=GRAPH 查看详细的拓扑检测结果，必要时可作为反馈信息提供给 NCCL 团队。

    性能调优 —— NCCL 会基于自动拓扑检测进行调优，无需用户手动配置。在某些基于 socket 的系统中，可以尝试调节 NCCL_SOCKET_NTHREADS 和 NCCL_NSOCKS_PERTHREAD 以提升网络带宽。NCCL 已针对部分云厂商（例如 AWS、GCP）预调优这两个变量。

    完整的 NCCL 环境变量列表请参考 NVIDIA NCCL 官方文档。

    你还可以通过 torch.distributed.ProcessGroupNCCL.NCCLConfig 和 torch.distributed.ProcessGroupNCCL.Options 进一步调节 NCCL 通信器，可在解释器中使用 help（例如 help(torch.distributed.ProcessGroupNCCL.NCCLConfig)）查阅更详细的说明。

    (distributed-basics)=

    基础概念

    torch.distributed 为多进程并行提供通信原语，可运行在一台或多台机器上。 {func}torch.nn.parallel.DistributedDataParallel 基于这些原语实现同步分布式训练， 可以封装任意 PyTorch 模型。与 {doc}multiprocessing 或 {func}torch.nn.DataParallel 不同，该模块支持多台网络互联的机器，同时需要显式为每个 进程启动一份主训练脚本。

    在单机同步场景下，torch.distributed 或 {func}torch.nn.parallel.DistributedDataParallel 相比其他数据并行方式（例如 {func}torch.nn.DataParallel）依然具备优势：

    • 每个进程维护自己的优化器，并在每次迭代执行完整的优化步骤。虽然梯度已经在各进程间聚合和平均， 但这种方式无需额外的参数广播，从而减少节点之间的张量传输。
    • 每个进程拥有独立的 Python 解释器，避免了单个 Python 进程驱动多个执行线程、模型副本或 GPU 时的额外开销和 GIL 争用。 这对于大量使用 Python 运行时的模型尤为重要，例如包含循环层或大量小模块的模型。

    初始化

    在调用其他方法之前，需要先调用 {func}torch.distributed.init_process_group 或 {func}torch.distributed.device_mesh.init_device_mesh 进行初始化。两者都会阻塞，直到所有进程加入。

    :::{warning} 初始化不是线程安全的。创建进程组时必须在单线程中进行，避免因 UUID 分配不一致或初始化过程中的竞争条件而导致阻塞。 :::

    目前支持三种初始化方式：

    TCP 初始化

    使用 TCP 初始化需要一个所有进程都可访问的网络地址和期望的 world_size。 第一种方式需要指定 rank 0 进程的地址，并且所有进程的 rank 必须手动指定。

    请注意，最新版本的分布式包不再支持多播地址，group_name 也已弃用。

    import torch.distributed as dist

# 使用任意一台机器的地址
dist.init_process_group(backend, init_method='tcp://10.1.1.20:23456',
                        rank=args.rank, world_size=4)

    共享文件系统初始化

    另一种初始化方法依赖所有机器都可访问的共享文件系统，并结合期望的 world_size。URL 需要以 file:// 开头，并指向共享文件系统上一个不存在的文件（所在目录必须存在）。 若文件不存在，初始化过程会自动创建，但不会自动删除，因此在下一次调用 {func}init_process_group 之前需要自行清理。

    同样地，自动分配 rank 的方式已经被移除，group_name 也已弃用。

    :::{warning} 该方法依赖文件系统支持 fcntl 锁。大多数本地文件系统和 NFS 都支持。 :::

    :::{warning} 此方法会在初始化时创建文件，并尽力在程序结束时删除。也就是说，每次使用文件系统初始化都需要一个全新的空文件。 如果重复使用上一轮遗留的文件（未能成功删除），很容易导致死锁或失败。因此，即使自动删除失败，用户也必须确保文件在训练结束时被清理， 尤其是在同一个文件名上多次调用 {func}init_process_group 时。规则很简单：每次调用前都要确保文件不存在或为空。 :::

    import torch.distributed as dist

# 必须显式传入 rank
dist.init_process_group(backend, init_method='file:///mnt/nfs/sharedfile',
                        world_size=4, rank=args.rank)

    环境变量初始化

    这种方式通过读取环境变量获取配置信息，方便用户自定义信息来源。需要设置的变量有：

    • MASTER_PORT —— 必填，为 rank 0 机器上的一个空闲端口；
    • MASTER_ADDR —— 必填（rank 0 可省略），为 rank 0 节点的地址；
    • WORLD_SIZE —— 必填，可以在环境变量或初始化函数中指定；
    • RANK —— 必填，可以在环境变量或初始化函数中指定。

    rank 0 节点会用来建立所有连接。该方式是默认选项，即 init_method 可省略或设为 env://。

    缩短初始化时间

    • TORCH_GLOO_LAZY_INIT —— 按需建立连接，而不是一次性构建全互连拓扑，对于非 all2all 操作可显著缩短初始化时间。

    关闭

    训练结束时务必调用 {func}destroy_process_group 释放资源。

    最简单的方式是在脚本中所有通信都完成后（通常是 main() 接近尾部）调用一次 {func}destroy_process_group()，并保持 group 参数为默认值 None。每个训练进程调用一次即可， 无需在启动器进程中调用。

    如果应用中存在多个进程组（例如 N 维并行），且不是所有 rank 都在超时时间内调用 {func}destroy_process_group，程序可能会在退出时挂起。这是因为 ProcessGroupNCCL 的析构函数内部会调用 ncclCommAbort，而该函数需要全体成员统一调用。 Python GC 的析构顺序不可预测，因此显式调用 {func}destroy_process_group 能保证 各个 rank 上的 ncclCommAbort 调用顺序一致，从而避免在析构函数中触发。

    重新初始化

    destroy_process_group 也可以用于销毁单个进程组。例如容错训练场景中，可能需要在运行时销毁旧的进程组并创建新的。 此时必须确保在调用 destroy 之后、重新初始化之前使用 torch.distributed 以外的方式对训练进程进行同步。 该行为目前尚未得到充分验证，属于已知限制。如有相关需求请在 GitHub 提交 issue 或 RFC。

    分组（Groups）

    默认情况下，集合通信运行在默认组（也称 world 上），并要求所有进程进入调用。 某些工作负载需要更细粒度的通信，这时可以通过 {func}~torch.distributed.new_group 创建包含任意进程子集的新分组。该函数返回一个不透明的组句柄，可在集合通信函数中通过 group 参数传入（集合通信即是按照常见模式交换信息的分布式函数）。

    同步与异步集合通信

    所有集合通信函数都支持以下两种模式，通过 async_op 参数进行控制：

    同步模式 —— 默认模式，即 async_op=False。函数返回时保证集合通信已执行完成。 对于 CPU 集合通信，可立即安全地使用输出结果。对于 CUDA 集合通信，由于 CUDA 操作是异步的，函数返回并不代表 CUDA 指令已经完成；在同一 CUDA stream 上使用输出结果是安全的。 若需要跨 stream 使用输出，用户必须自行同步。有关 CUDA 语义和 stream 同步的更多说明，参见 CUDA 语义。CPU 与 CUDA 行为的差异可参考下方示例脚本。

    异步模式 —— 当 async_op=True 时，集合通信函数返回一个分布式请求对象。该对象同样提供 is_completed() 与 wait() 方法：

    • is_completed() —— 对于 CPU 操作，若完成则返回 True；对于 CUDA 操作， 只有当底层 CUDA 指令完成时才返回 True。此外，该调用不会自动触发同步。
    • wait() —— 阻塞当前进程直到操作完成。is_completed() 在 wait() 返回后一定为 True。

    异步接口允许与其他计算重叠，或在单线程中顺序调度多个集合通信。

    集合通信的性能分析

    可以使用 torch.profiler（推荐，1.8.1 及以上版本提供）或 torch.autograd.profiler 对集合通信和点对点通信接口进行性能分析。所有内置后端（gloo、nccl、mpi） 均受到支持，相关调用会在分析结果/日志中按预期展示。使用方式与分析普通 torch 运算相同：

    import torch
import torch.distributed as dist
with torch.profiler():
    tensor = torch.randn(20, 10)
    dist.all_reduce(tensor)

    更多 profiler 功能请参阅 Profiler 文档。

    对象级集合通信

    :::{warning} 对象集合通信存在若干严重限制，请继续阅读以确认是否适用于你的场景。 :::

    对象集合通信可在能被 pickle 的任意 Python 对象上执行集合模式（如广播、all_gather 等）。 其典型流程如下：

    1. 将输入对象 pickle 成字节串并写入字节张量；
    2. 通过一次集合通信告知其它进程该张量的大小；
    3. 根据大小分配合适的张量；
    4. 再执行一次集合通信传输对象数据；
    5. 将字节串反序列化回 Python 对象。

    对象集合通信可能带来令人意外的性能或内存开销，应谨慎使用。常见问题包括：

    序列化/反序列化耗时不均衡 —— 序列化速度取决于对象数量、类型和大小。 例如在 gather_object 中，接收端需要反序列化 N 倍于发送端的对象数量，可能导致其它进程在后续集合通信中超时。

    低效的张量通信 —— 建议使用常规集合通信 API 处理张量，而不是对象集合通信。 如果通过对象集合发送张量，会经历序列化与反序列化（非 CPU 张量还需额外的 CPU 同步与设备到主机的复制）， 几乎所有场景下都值得改写代码，直接调用张量集合通信接口。

    设备不一致的问题 —— 如果仍要通过对象集合发送张量，还需注意 CUDA（或其他加速器）设备。 当 pickle 一个位于 cuda:3 的张量并在另一进程中 unpickle 时，得到的张量依旧位于 cuda:3， 而不会根据本地默认设备调整。常规张量集合通信的输出张量始终位于本地设备，更符合预期。

    此外，如果首次在进程中 unpickle 张量，会隐式创建一个 CUDA 上下文，可能浪费大量显存。 可在传入对象集合前将张量移动到 CPU 以规避该问题。

    第三方后端

    除了内置的 GLOO/MPI/NCCL 后端之外，PyTorch 分布式支持通过运行时注册机制接入第三方后端。 如需通过 C++ 扩展开发第三方后端，请参考 自定义 C++/CUDA 扩展教程 以及 test/cpp_extensions/cpp_c10d_extension.cpp。第三方后端的能力取决于自身的实现。
