    分布式数据并行（DDP）

    实现版本

    本文基于 PyTorch 1.4 中 torch.nn.parallel.DistributedDataParallel 的实现细节撰写。随着版本演进，底层实现可能发生变化，逻辑或参数需以实际代码为准。

    torch.nn.parallel.DistributedDataParallel（DDP）可以透明地执行分布式数据并行训练。本篇以示例入手，逐步剖析一次迭代在内部经历的关键步骤与组件。

    示例

    以下示例展示如何使用一层 torch.nn.Linear 构建本地模型，包裹 DDP 后完成一次前向、反向与优化步骤。执行完毕后，各进程上的模型参数保持完全一致。

    import os
import torch
import torch.distributed as dist
import torch.multiprocessing as mp
import torch.nn as nn
import torch.optim as optim
from torch.nn.parallel import DistributedDataParallel as DDP


def example(rank, world_size):
    # 创建默认进程组
    dist.init_process_group("gloo", rank=rank, world_size=world_size)
    # 本地模型
    model = nn.Linear(10, 10).to(rank)
    # 构建 DDP 包装
    ddp_model = DDP(model, device_ids=[rank])
    # 优化器与损失
    loss_fn = nn.MSELoss()
    optimizer = optim.SGD(ddp_model.parameters(), lr=0.001)

    outputs = ddp_model(torch.randn(20, 10).to(rank))
    labels = torch.randn(20, 10).to(rank)
    loss_fn(outputs, labels).backward()
    optimizer.step()


def main():
    world_size = 2
    mp.spawn(example, args=(world_size,), nprocs=world_size, join=True)


if __name__ == "__main__":
    # c10d 默认 "env" 初始化方式需要的环境变量
    os.environ["MASTER_ADDR"] = "localhost"
    os.environ["MASTER_PORT"] = "29500"
    main()

    DDP 与 TorchDynamo 兼容。若希望配合 TorchDynamo 使用，请先包裹 DDP，再调用 torch.compile，以便 TorchDynamo 能基于 DDP 的 bucket 大小应用 DDPOptimizer（更多细节见 TorchDynamo 文档）。

    ddp_model = DDP(model, device_ids=[rank])
ddp_model = torch.compile(ddp_model)

    内部设计

    以下按迭代顺序解释 DDP 在每一步内部的工作原理。

    • 前置条件：DDP 依赖 c10d ProcessGroup 执行通信操作，因此在创建 DDP 前必须先初始化对应的进程组。
    • 构造阶段：DDP 构造函数会将 rank 0 的 state_dict() 广播到其余进程，确保各副本状态一致。随后在本地创建 Reducer，用于在反向传播阶段同步梯度。Reducer 会依据 bucket_cap_mb 与参数大小将梯度分桶，以减少通信次数；桶的分配顺序与 model.parameters() 的逆序大致一致，以匹配梯度触发顺序。构造过程中还会为每个参数注册 autograd hook，以便梯度就绪时触发通信。
    • 前向阶段：DDP 将输入转发给本地模型。若设置 find_unused_parameters=True，DDP 会遍历 autograd 图，标记在本次反向中不会用到的参数，防止后续等待不存在的梯度（遍历会带来额外开销，建议仅在必要时启用）。
    • 反向阶段：loss.backward() 由用户显式调用，DDP 依赖之前注册的 hook 感知梯度就绪。当桶内所有梯度准备完毕时，Reducer 会启动异步 allreduce，计算跨进程的平均梯度；待所有桶完成后，将平均结果写回对应参数的 grad 字段。
    • 优化阶段：优化器依然只看到本地模型，但由于起始状态一致且每轮梯度相同，各副本在更新后保持同步。

    AllReduce 顺序要求

    所有进程上的 Reducer 必须以完全相同的顺序调用 allreduce。DDP 通过始终按桶索引顺序发起通信来保证一致性，否则会出现错误结果或反向传播挂起。

    实现参考

    以下列出核心实现文件，便于进一步阅读源码：

    ProcessGroup

    • ProcessGroup.hpp：定义了进程组抽象接口，内置 ProcessGroupGloo、ProcessGroupNCCL、ProcessGroupMPI。DDP 在初始化阶段调用 ProcessGroup::broadcast() 同步模型，在反向阶段调用 ProcessGroup::allreduce() 聚合梯度。
    • Store.hpp：为进程组提供 rendezvous 服务，帮助进程发现彼此。

    DistributedDataParallel

    • distributed.py：Python 入口，实现 DDP 的初始化与前向逻辑，并调用底层 C++ 模块。其 _sync_param 方法负责在单进程多设备场景下的参数同步，同时在前向前广播模型缓冲区。跨进程的梯度同步则由 Reducer.cpp 负责。
    • comm.h：实现合并广播的辅助函数，用于初始化时广播模型状态、前向前同步缓冲区。
    • reducer.h：核心梯度同步逻辑所在，主要入口包括：
    • Reducer 构造函数：在 distributed.py 中被调用，注册 Reducer::autograd_hook()；
    • autograd_hook()：由 autograd 引擎在梯度就绪时触发；
    • prepare_for_backward()：在 DDP 前向结束时调用，用于在 find_unused_parameters=True 时遍历 autograd 图并标记未使用参数。

    TorchDynamo 中的 DDPOptimizer

    DDP 的性能优势来自反向阶段将 allreduce 与计算重叠。然而当 TorchDynamo 搭配 AOTAutograd 编译整段前向与反向图时，allreduce 会在优化过的反向计算结束后才通过 autograd hook 启动，导致两者无法重叠。

    TorchDynamo 的 DDPOptimizer 会按照 DDP bucket 的逻辑边界切分前向图，再对每个片段分别执行 AOTAutograd 与编译。这样就能在反向过程中穿插触发 allreduce，恢复通信与计算的重叠。

    • 更多介绍与实验见官方博客。
    • 代码参考：torch/_dynamo/backends/distributed.py。

    调试时可设置环境变量：

    • TORCH_LOGS="ddp_graphs"：输出包含完整图的日志；
    • TORCH_LOGS="dynamo,distributed,dist_ddp"：输出不含图的简洁日志，便于查看 bucket 切分；
    • torch._dynamo.config.optimize_ddp=False：禁用 DDPOptimizer，虽然仍可运行，但性能会有所下降。
