    分布式检查点 —— torch.distributed.checkpoint

    分布式检查点（Distributed Checkpoint，简称 DCP）支持多个 rank 并行地保存和加载模型， 同时可以在加载阶段自动重新切分张量，从而允许在一个集群拓扑中保存、在另一个拓扑中恢复。

    DCP 与 torch.save / torch.load 的主要差异包括：

    • 生成的检查点包含多个文件，至少每个 rank 一个文件；
    • 采用就地操作，模型需要先分配好数据缓冲区，DCP 会复用这些存储空间。

    由于历史原因，FSDP 和 DDP 的 state_dict 可能拥有不同的键（例如 layer1.weight）， 即便原始未并行的模型完全一致。此外，FSDP 支持多种模型 state_dict（完整版本、分片版本等）。 优化器的 state_dict 则使用参数 ID 而非全限定名称，当结合多种并行方式（例如流水线并行）时，这可能带来额外问题。

    为解决这些挑战，我们提供了一组方便管理 state_dict 的 API。get_model_state_dict() 返回的模型 state_dict 键名与未并行模型保持一致；类似地，get_optimizer_state_dict() 返回的优化器 state_dict 对所有并行方式均使用一致的键名。get_optimizer_state_dict() 会将参数 ID 转换为与未并行模型 state_dict 相同的全限定名称。使用这些 API 得到的结果可直接传递给 torch.distributed.checkpoint.save() 与 torch.distributed.checkpoint.load()，无需额外转换。

    set_model_state_dict() 和 set_optimizer_state_dict() 则用于加载由上述 getter API 生成的 state_dict。
